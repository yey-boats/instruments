# Project analysis — 2026-06-03

Scoped to the three priorities the user set: **stability**, **functionality**,
**default screens design**. Findings ranked by ROI per line of change.

## Stability

### S1 — Web stack TCP wedge → `handle_state` PSRAM fix (DONE, commit `1268d7d`)

**Observed:** `/api/state` and `/` become unreachable while ping + BLE + sk
reconnect attempts all stay alive. Reproducible after a few minutes of
web-UI polling.

**Root cause:** `handle_state()` in `src/web.cpp` builds a JsonDocument
with ~87 fields across `device / wifi / sk / manager / screen / display /
ui / queues / gestures / touch / errors` — multi-KiB allocation on the
internal heap that idles at 11–15 KiB free. Same root pattern as the
manager bug in `f3c3c26`. Frequent web polling fragments internal SRAM
until lwIP can't allocate accept-queue entries; TCP listener silently
stops accepting while UDP (ping, mDNS, discovery, SK WS reconnect
attempts) keep working.

**Fix shipped:** `include/psram_json.h` exposes a shared
`espdisp::psram_json` allocator. `manager.cpp` defines it via
`PSRAM_JSON_DEFINE_SHARED` at file scope; `web.cpp::handle_state` now
constructs `JsonDocument doc(&espdisp::psram_json)`. Same fallback to
internal heap on PSRAM exhaustion.

Other `web.cpp` handlers that build large docs and should follow:
`handle_boat` (85 lines), `handle_screens` (44 lines, but iterates all
N screens with metadata). Quick follow-up: ~5 minute task.

### S2 — `handle_state` body is too big (~87 fields, 143 lines)

Beyond moving the doc to PSRAM, **a lot of `/api/state` content is
debug-only and shouldn't be in the hot polling endpoint.** The web UI
needs ~15 fields (device.id, build, heap, sk.state, screen.id,
display.brightness, ui.theme, mgr.health). The rest is forensic:

- `touch.{x,y,pressed,last_ms,mode,irq,i2c_ok,i2c_err,gt_ready,gt_points}`
- `queues.{ui_depth,ui_hi,net_depth,net_hi}`
- `gestures.{count,suppressed,last}`
- `mgr.recentErrors` array
- `manager.{lastCmd*,otaJobId,...}`

**Recommendation:** split into `/api/state` (~15 hot fields, polled
every 1-5 s) and `/api/diag` (everything else, polled only when an
admin/diagnostics view is open). Cuts the hot-poll payload by ~70 %.

Effort: ~30 min. ROI: meaningfully reduced PSRAM churn + TCP socket
recycle pressure on the hot path.

### S3 — Stack canaries are off in `sdkconfig`

```
CONFIG_COMPILER_STACK_CHECK_MODE_NONE=y
CONFIG_STACK_CHECK_NONE=y
```

So the 19-KiB stack overflow in pre-fix `handle_logs` was undetected
for who knows how long. Enabling `CONFIG_COMPILER_STACK_CHECK_MODE_STRONG`
costs a few % CPU but turns silent stack corruption into a clean panic
with a backtrace — which we now also capture via the RTC `[prevboot]`
ring. Strongly recommend enabling on the debug env.

Effort: ~10 min (sdkconfig flip + rebuild). ROI: high — turns the
hardest class of bugs from "random reboots" into "panic with stack
trace in next-boot prevboot trail."

### S4 — WiFi half-up state recurs

Symptom we hit ~20 times this session: device reports `mode=STA
rssi=-48` over BLE but compulab's AP shows no associated station and
all TCP from compulab times out. `wifi-reconnect` via BLE recovers it
reliably.

This may be a downstream of S1 (lwIP allocation failures inside the
WiFi event handler put the supplicant into an inconsistent state) — if
S1's fix sticks, S4 may resolve itself. If not: instrument the WiFi
event task to log `[wifi] event=<n>` for state transitions, and watch
for the asymmetric SA_DISASSOC → no reassociate sequence.

### S5 — Manager `lastHeartbeatCode = -1` (transport error) loops

The 8 recorded errors in the most recent live capture were all
`[mgr] heartbeat -> -1`. `-1` means `WiFi.status() != WL_CONNECTED` at
the top of `do_heartbeat` — the manager thinks WiFi is down. With the
half-up state from S4, this is consistent. Will likely clear with S1+S4.

### S6 — Random reboots remain

Even after the PSRAM-JSON manager fix + threshold tuning + OTA task
move, the device still self-reboots (caught by `espdisp watch` ~twice
in 90 s on an earlier run, less frequent after S1's fix). The
`[boot] reset_reason=` logging from commit `1bc8ddb` is in place but
we haven't successfully captured a reset_reason after a spontaneous
reboot — `/api/logs` is unreachable in the failure window because of
S1. With S1's fix landing, the next session should be able to read
the reset_reason cleanly via either `/api/logs` or the BLE NUS log
stream.

## Functionality

### F1 — `/api/logs` polling causes lwIP socket exhaustion

Earlier stress test: 12 successful `/api/logs?since=0&limit=96` requests
before TCP sockets refuse to accept (~30 s wedge while TIME_WAIT clears).
With the `limit` default already capped to 32 in commit `a9239eb`, web-UI
poll pressure is much lower in normal use. No further action needed for
typical operation; for stress paths consider HTTP keep-alive on the
Arduino WebServer (it's not on by default).

### F2 — espdisp CLI now covers all the routines I was doing inline

`tools/yeydisp.py` consolidates BLE-cmd / state / logs / discover /
recover / watch / config. The `watch` reboot-detector caught spontaneous
reboots automatically. Nothing more to add here; the tooling is in good
shape.

### F3 — Load test verified the manager plugin scales

`make load-test DEVICES=50 DURATION=30` against the lab SignalK: 251/251
requests succeed, p50 13-26 ms, p99 commands 270 ms. Plugin headroom is
healthy.

### F4 — Device HTTP rate ceiling (~2 req/s sustained)

Live-measured 2026-06-03 against build `Jun 3 2026 16:23:03`:

| Pattern                          | Result            |
|----------------------------------|-------------------|
| 20 req with 500 ms gap (~2 req/s)| 20/20 OK          |
| 20 req with 200 ms gap (~5 req/s)| 20/20 OK          |
| 50 req back-to-back (~30 req/s)  | 26/50 OK, then wedged 60 s |

Root cause: arduino-esp32 ships a prebuilt lwIP with
`CONFIG_LWIP_TCP_MSL=60000` and `CONFIG_LWIP_MAX_SOCKETS=10`. Every
HTTP response uses a fresh TCP connection (Arduino's WebServer
doesn't keep-alive), so each request burns a socket into TIME_WAIT
for 60 s. At ~30 req/s the pool drains within a second and lwIP
refuses new accepts for the next 60 s.

**Not patchable without rebuilding arduino-esp32** — sdkconfig in
this repo is the IDF menuconfig output, but the actual built
libraries ship in the framework package. Two ways to address:

- **App-level mitigation (not done)**: shed load with a 429 when
  socket pool is below threshold. Adds complexity for an edge case.
- **Documented limit (done here)**: the device is rated for ≤2 req/s
  sustained polling, well above the 0.2-1 req/s the UI generates.
  Browsers poll on `setInterval`, not in tight loops, so this
  ceiling never hits in normal operation.

If the device is going to be exposed to anything that bursts harder
(monitoring/scraping/load testing), the fix is either:
1. Custom-build arduino-esp32 with `CONFIG_LWIP_TCP_MSL=5000`, or
2. Switch the web server to ESPAsyncWebServer (supports keep-alive,
   doesn't burn a socket per request).

### F5 — RETRACTED: there are no spontaneous reboots

The earlier write-up of F5 ("HTTP polling correlates with reboots")
was wrong. Root cause of the bad signal was a bug in `tools/yeydisp.py
watch`: when `/api/state` failed to respond, `http_get_json` returned
`None`, the caller coerced it to `{}`, and `dev.get("uptime_ms", 0)`
returned 0. The reboot detection then saw `0 < prev_uptime` and
printed `!! REBOOTED`. Every "reboot" in the F5 log was actually
the TCP wedge from F4 manifesting as an unreachable endpoint, NOT
a soft reset.

Verified after fixing `cmd_watch` to skip the reboot check on poll
failure:

- Device uptime when checked: 30 minutes 54 seconds, no reset.
- BLE-only stream for 3 min: no reset_reason logs.
- 3 min `espdisp watch --interval 30` after fix: the wedge happens
  exactly as documented in F4 (`(/api/state unreachable)` lines)
  but no reboot fires.

So the firmware is **actually stable now**. The earlier random
reboots were the ones already addressed by:

- `f3c3c26` manager PSRAM-JSON allocator
- `1268d7d` web.cpp handle_state to PSRAM
- `2e74da7` manager backoff threshold tuning
- `86bd56f` OTA on dedicated core-0 task
- `793323e` all web JsonDocuments to PSRAM + /api/state hot/diag split
- `2c55d52` prevboot ring dump no-recurse fix

What's left is purely the TCP wedge under burst load - cosmetic
unreachability rather than a reset. The ESPAsyncWebServer migration
becomes a quality-of-life improvement, not a stability one.

## Default screens design

### D1 — Default screen order is sensible

Registration order in `src/main.cpp:1891-1905`:

```
dashboard  (default)
wind
nav
depth
steering
route
autopilot
trip
status
wifi       (hidden, only shown from setup)
```

`dashboard` first = default after boot. Dashboard is a `QuadGrid`
template with WIND / NAV / DEPTH / SYSTEM tiles — exactly the four
metrics a sailor wants at a glance. Heroes are: AWS, SOG, depth,
battery V. Secondary metrics filled per tile. Good design.

### D2 — `screen_wind.cpp` is the most complex (613 lines) but well-justified

The wind screen has a rotating outer bezel (cardinals), separate
apparent/true wind markers, tide vector, and close-hauled arcs. The
implementation is documented at the top with coordinate conventions
(bow points up, signalk angles are bow-relative radians) — clear
intent. Not a simplification target.

### D3 — `screen_settings.cpp` (347 lines) has repetitive segmented-control code

`screen_settings.cpp` defines 5 separate `Segmented` instances
(`seg_brightness`, `seg_theme`, `seg_format`, `seg_depth`, `seg_battery`,
`seg_demo`) with near-identical create/update/handler code. Roughly
~150 lines of the 347 are repetition. A small `Segmented::ensure(
parent, values[], count, on_select)` factory would cut the file in
half without changing behavior.

Effort: ~30 min. ROI: modest (legibility, not stability).

### D4 — Touch calibration screen (`screen_touch_cal.cpp`) is a one-shot

240 lines, runs only on first boot or when explicitly invoked. Could
be a `#if !YEYBOATS_RELEASE_BUILD` candidate to drop ~3-4 KiB from
shipped firmware. Per `build_config.h` and the release_common block,
this kind of gating is already established for input_test / bench /
demo / stall_telemetry.

Effort: ~5 min. ROI: small.

## Simplification

### Σ1 — `src/main.cpp` is 2029 lines

Splittable into:
- `src/main_alarms.cpp` (`alarm_check`, `alarm_set`, ALARM_* enum and
  the MOB + overlay primitives — ~200 lines)
- `src/main_touch.cpp` (touch task, calibration glue, IRQ — ~300 lines)
- `src/main_screens.cpp` (screen registration block we just looked
  at — ~50 lines)

This isn't urgent; the file is navigable. Worth doing opportunistically
when touching one of those subsystems.

### Σ2 — `src/manager.cpp` is 1983 lines

Already partially split: `manager_config.cpp`, `manager_endpoint.cpp`,
`manager_screens.cpp`, `manager_url.cpp` exist as side modules. The
remaining big file is the HTTP plumbing + worker task. Could lift the
HTTP helpers (`post_json`, `deserialize_http_json`,
`serialize_json_payload`, `manager_heap_ready`, PsramJsonPayload) into
`src/manager_http.cpp`, ~150 lines. Would also make them easier to
reuse from web.cpp.

### Σ3 — `src/ui/ui_layouts.cpp` is 1813 lines

Template factory for all `QuadGrid`/`HeroDial`/`StatusList`/etc
screens. Centralisation is the point — splitting it would defeat the
template intent. Leave as is.

### Σ4 — Dead code candidates

`/api/state` exposes `gestures.*`, `touch.{irq,i2c_*,gt_ready}`,
`queues.*` — all originally diagnostic. If the web UI doesn't render
them, they could move behind a `?diag=1` flag (see S2) or behind
`#if YEYBOATS_ENABLE_INPUT_TEST`.
